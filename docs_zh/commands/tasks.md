---
description: 基于可用设计工件生成可操作的、依赖排序的 tasks.md
---

用户输入可以通过 agent 直接提供或作为命令参数提供 - 在继续执行提示之前，你**必须**考虑它（如果不为空）。

用户输入：

$ARGUMENTS

1. 从仓库根目录运行 `.specify/scripts/bash/check-prerequisites.sh --json` 并解析 FEATURE_DIR 和 AVAILABLE_DOCS 列表。所有路径必须是绝对的。
2. 加载并分析可用的设计文档：
   - 始终读取 plan.md 获取技术栈和库
   - **如果存在**：读取 data-model.md 获取实体
   - **如果存在**：读取 contracts/ 获取 API 端点
   - **如果存在**：读取 research.md 获取技术决策
   - **如果存在**：读取 quickstart.md 获取测试场景

   注意：并非所有项目都有所有文档。例如：
   - CLI 工具可能没有 contracts/
   - 简单的库可能不需要 data-model.md
   - 基于可用的内容生成任务

3. 按照模板生成任务：
   - 使用 `.specify/templates/tasks-template.md` 作为基础
   - 用基于以下内容的实际任务替换示例任务：
     * **设置任务**：项目初始化、依赖项、linting
     * **测试任务 [P]**：每个合约一个，每个集成场景一个
     * **核心任务**：每个实体、服务、CLI 命令、端点一个
     * **集成任务**：数据库连接、中间件、日志
     * **完善任务 [P]**：单元测试、性能、文档

4. 任务生成规则：
   - 每个合约文件 → 标记为 [P] 的合约测试任务
   - data-model 中的每个实体 → 标记为 [P] 的模型创建任务
   - 每个端点 → 实现任务（如果共享文件则不是并行）
   - 每个用户故事 → 标记为 [P] 的集成测试
   - 不同文件 = 可以并行 [P]
   - 相同文件 = 顺序（无 [P]）

5. 按依赖关系排序任务：
   - 设置优先于一切
   - 实现前测试（TDD）
   - 服务前模型
   - 端点前服务
   - 集成前核心
   - 完善前一切

6. 包含并行执行示例：
   - 可以一起运行的 [P] 任务组
   - 显示实际的 Task agent 命令

7. 创建 FEATURE_DIR/tasks.md，包含：
   - 来自实现计划的正确功能名称
   - 编号任务（T001、T002 等）
   - 每个任务的清晰文件路径
   - 依赖说明
   - 并行执行指导

任务生成的上下文：$ARGUMENTS

tasks.md 应该立即可执行 - 每个任务必须足够具体，以便 LLM 可以在没有额外上下文的情况下完成它。
